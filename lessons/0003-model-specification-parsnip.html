Specifying Models with {parsnip}

One interface for every algorithm — swap engines without rewriting your code

The Problem with Base R Modeling

You've probably felt this: logistic regression uses glm(..., family = binomial), random forests need randomForest() or ranger() with completely different argument names, and regularized regression wants glmnet() with a matrix input instead of a data frame. Every package speaks a different language.

This forces a painful choice: commit to one algorithm early (because switching means rewriting everything), or spend your time wrangling syntax instead of thinking about the model. Neither is good when you're trying to answer a business question about attrition.

{parsnip} solves this with a single consistent interface. You declare what kind of model you want, which package should run it, and what type of outcome you're predicting — three decisions, three function calls, same syntax every time.

The Three Components

Model Type
logistic_reg()
The mathematical structure
Engine
set_engine("glm")
The R package that runs it
Mode
set_mode("classification")
Category or number outcome

These three things are declared independently of your data. The specification is just a description — nothing is computed until you call fit().

Your First Attrition Model Specification

library(tidymodels)

# Logistic regression — natural starting point for attrition (Yes/No outcome)
lr_spec <- logistic_reg() %>%
  set_engine("glm") %>%
  set_mode("classification")

lr_spec
#> Logistic Regression Model Specification (classification)
#> Computational engine: glm

Nothing has been computed yet. You've described the model — not run it. That separation lets you swap engines or compare algorithms without touching anything downstream.

Fitting the Model

# Fit to the training data from Lesson 2
lr_fit <- lr_spec %>%
  fit(Attrition ~ ., data = attr_train)

lr_fit

The dot in the formula means "use all other columns as predictors." In a real analysis you'd be more selective, but this gives you a working model immediately to build on.

The Power of Swapping Engines

Here is the key insight of parsnip. To switch algorithms, you change two lines. Everything downstream — fitting, predicting, evaluating — stays identical.

Choose a model for attrition prediction

The fit(), predict(), and evaluation calls you write once work with all three. This is why you never need to commit to an algorithm early in tidymodels.

Making Predictions

All parsnip models return predictions as a tibble with predictable column names — making it trivial to feed results into any tidyverse operation.

# Class predictions — who is predicted to leave?
predict(lr_fit, new_data = attr_test)
#> # A tibble: 294 × 1
#>   .pred_class
#> 1 No
#> 2 No  ...

# Probability scores — how confident is the model?
predict(lr_fit, new_data = attr_test, type = "prob")
#> # A tibble: 294 × 2
#>   .pred_No .pred_Yes
#>      0.943     0.057
#>      0.891     0.109  ...

The .pred_Yes column is your attrition risk score — the number you'd show an HR business partner when flagging employees at elevated risk of leaving.

People Analytics Note For attrition presentations, probabilities (type = "prob") are almost always more useful than class predictions. A score of 0.72 tells a richer story than "Yes." It lets leadership ask: "Should we act above 0.5? 0.7?" — that threshold decision belongs to the business, not the model.

What's Still Missing

You now have a model spec and can fit and predict. But there's a gap: preprocessing — encoding categorical columns, normalizing numerics, handling missing values — isn't part of the model spec. That belongs in a {recipes} recipe, and the two need to travel together so the same transformations always apply to new data. That's exactly what {workflows} solves — Lessons 4 and 5.
