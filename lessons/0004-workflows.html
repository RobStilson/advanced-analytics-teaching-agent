Bundling with {workflows}

Keeping preprocessing and model together — forever

The Preprocessing Amnesia Problem

Imagine you built an attrition model six months ago. You mean-centered the numeric features, one-hot encoded the job role column, and removed near-zero-variance predictors. Now HR wants you to score 200 new employees hired since then.

Do you remember every preprocessing step? Do you have the training-set means you used for centering — not the new hires' means? Did you apply the steps in the right order?

This is preprocessing amnesia — and it's one of the most common sources of silent errors in People Analytics. The model and the preprocessing steps live in different places and quietly drift apart.

The Silent Error Preprocessing new data with its own statistics instead of training-set statistics doesn't throw an error. Your predictions just become subtly wrong — and you won't notice until someone asks why the model "isn't working anymore."

What a Workflow Is

A workflow is a container that holds a preprocessor and a model specification together as one object. When you fit it on training data, it learns the preprocessing parameters. When you call predict(), it automatically applies the exact same preprocessing — using training-set statistics — to whatever new data you pass in.

Inside a workflow
Preprocessor (formula or recipe)
+
Model specification (parsnip)
↓ fit() on training data
Fitted workflow — preprocessing params + trained model, bundled

The fitted workflow is a single object you can save, share, or hand to a colleague. Scoring new data is one line. No manual preprocessing required.

Building a Workflow, Step by Step

Start with an empty container
workflow() creates the bundle. It holds nothing yet.
library(tidymodels) attr_wf <- workflow()
Add the model specification
Drop in the parsnip spec from Lesson 3. The workflow now knows what algorithm to use.
lr_spec <- logistic_reg() %>% set_engine("glm") %>% set_mode("classification") attr_wf <- workflow() %>% add_model(lr_spec)
Add a preprocessor
add_formula() is the simple option — it tells the workflow which columns to use. In Lesson 5, you'll swap this for add_recipe(), which handles encoding, imputation, and more.
attr_wf <- workflow() %>% add_model(lr_spec) %>% add_formula(Attrition ~ .) attr_wf #> ══ Workflow ══════════════════════════ #> Preprocessor: Formula #> Model: logistic_reg() #> #> ── Preprocessor ────────────────────── #> Attrition ~ . #> #> ── Model ───────────────────────────── #> Logistic Regression Model Specification (classification) #> Computational engine: glm
Fit — one call trains everything
fit() applies the preprocessor to training data, then trains the model. Both are locked inside the fitted workflow.
attr_fit <- attr_wf %>% fit(data = attr_train) attr_fit #> ══ Workflow [trained] ════════════════ #> Preprocessor: Formula #> Model: logistic_reg() #> #> ── Model ───────────────────────────── #> Coefficients: #> (Intercept) Age BusinessTravel ... #> 2.341 -0.031 0.421 ...
Predict — preprocessing is automatic
Pass raw data. The fitted workflow handles all preprocessing using training-set parameters. Nothing manual required — now or in six months.
# Score test set employees predict(attr_fit, new_data = attr_test, type = "prob") #> # A tibble: 294 × 2 #> .pred_No .pred_Yes #> 0.943 0.057 #> 0.891 0.109 ... # Same call works for new hires next year predict(attr_fit, new_data = new_hires_2027, type = "prob")

Updating a Workflow

update_model() lets you swap in a different model spec without rebuilding anything else — the same fair comparison idea from Lesson 3, now formalized:

# Swap logistic regression for random forest — one line
rf_spec <- rand_forest(trees = 500) %>%
  set_engine("ranger") %>%
  set_mode("classification")

attr_rf_fit <- attr_wf %>%
  update_model(rf_spec) %>%
  fit(data = attr_train)

Same formula, same training data, same predict() call — just a different model inside. This is how you compare algorithms fairly: identical preprocessing, identical data, only the model changes.

The Deliverable Save your fitted workflow with saveRDS(attr_fit, "attrition_model.rds"). Six months later: attr_fit <- readRDS("attrition_model.rds"), then predict(attr_fit, new_data = ...). The exact preprocessing from your original training run is baked in — no notes, no scripts, no guessing.

What's Coming: add_recipe()

This lesson used add_formula(), which handles column selection but nothing more. In Lesson 5 you'll replace it with add_recipe() — which handles everything HR data actually needs: encoding job role and department as dummy variables, imputing missing survey scores, and normalizing tenure and age. The workflow object stays identical; only the preprocessor grows.
