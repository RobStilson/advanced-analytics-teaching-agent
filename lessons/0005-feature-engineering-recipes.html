Feature Engineering with {recipes}

Transforming raw HR data into what models actually need

Why add_formula() Isn't Enough

The IBM HR Attrition dataset has 35 columns. Several are factors — JobRole has nine levels, Department has three, BusinessTravel has three. Most algorithms can't handle raw factor columns: they need numbers. And numeric columns like MonthlyIncome span from 1,009 to 19,999 — a scale that will dominate distance-based algorithms unless you normalize it.

add_formula(Attrition ~ .) from Lesson 4 tells the workflow which columns to use, but does none of this transformation work. That's the job of a recipe.

What HR Data Looks Like Coming In

ColumnTypeWhat needs to happen
AttritionoutcomeLeft alone — it's what we're predicting
JobRole, Department, EducationFieldcategoricalEncode as binary dummy variables
BusinessTravel, MaritalStatus, OverTimecategoricalEncode as binary dummy variables
MonthlyIncome, Age, YearsAtCompanynumericNormalize to mean 0, SD 1
EmployeeCount, StandardHoursnumericRemove — zero variance (same value for everyone)

The Four Steps Your Recipe Needs

Toggle each step to see what it does to the attrition data:

Recipe step explorer
step_zv(all_predictors())
Remove zero-variance columns — ones with the same value in every row
step_other(all_nominal_predictors(), threshold = 0.05)
Collapse rare factor levels (under 5% of rows) into "other" before encoding
step_dummy(all_nominal_predictors())
Convert each categorical column into binary 0/1 columns — one per level, minus one
step_normalize(all_numeric_predictors())
Center and scale all numeric predictors to mean 0, SD 1
Toggle steps above to see their effects on the attrition data.
Order Matters Always put step_other() before step_dummy(). If you run step_dummy() first, rare levels become their own binary columns — then step_other() has nothing left to collapse. Steps execute sequentially, top to bottom.

The Complete Attrition Recipe

library(tidymodels)

attr_recipe <- recipe(Attrition ~ ., data = attr_train) %>%
  # 1. Drop columns with zero variance
  step_zv(all_predictors()) %>%
  # 2. Collapse rare levels before encoding
  step_other(all_nominal_predictors(), threshold = 0.05) %>%
  # 3. Encode all factors as dummy variables
  step_dummy(all_nominal_predictors()) %>%
  # 4. Normalize all numeric predictors
  step_normalize(all_numeric_predictors())

attr_recipe

recipe() takes the training data to learn column types and levels — but transforms nothing yet. That happens when the recipe is fitted inside a workflow.

Plugging the Recipe into Your Workflow

One change from Lesson 4: swap add_formula() for add_recipe(). Everything else is identical:

attr_wf <- workflow() %>%
  add_recipe(attr_recipe) %>%   # replaces add_formula()
  add_model(lr_spec)

attr_fit <- attr_wf %>%
  fit(data = attr_train)

predict(attr_fit, new_data = attr_test, type = "prob")
What fit() Does Now When fit() runs on this workflow it: (1) drops constant columns, (2) identifies rare levels, (3) learns the dummy structure from training factor levels, (4) computes training-set mean and SD for each numeric column. All four sets of parameters are locked inside the fitted workflow — new data gets exactly the same treatment.

Inspecting What the Recipe Did

Outside of a workflow, prep() + bake() let you inspect the transformed data — useful for debugging:

# prep() estimates parameters using training data only
prepped <- prep(attr_recipe)

# bake() applies the fitted recipe — NULL means use training data
transformed <- bake(prepped, new_data = NULL)

glimpse(transformed)
# JobRole is now 8 binary columns: JobRole_Manager, JobRole_Research.Scientist ...
# MonthlyIncome is centered around 0
# EmployeeCount and StandardHours are gone
