Resampling & Cross-Validation

Getting an honest performance estimate without touching the test set

The Gap Between Now and Lesson 2

Back in Lesson 2 you split your data and set the test set aside. You've been building — a recipe, a parsnip spec, a workflow. But here's the problem: how do you know if your model is actually good before you look at the test set?

You can't measure performance on the training data — the model has already seen it, so the numbers are optimistically inflated. A random forest can nearly memorize training data and report near-perfect accuracy, while performing much worse on new employees. You can't look at the test set yet either — that would spend your one-shot evaluation prematurely.

Resampling solves this by repeatedly splitting the training set into mini train/assess pairs, fitting and evaluating the workflow each time, then averaging the results. You get an honest performance estimate using only training data — the test set stays untouched.

How 10-Fold Cross-Validation Works

The training set is divided into 10 roughly equal folds. In each of 10 iterations, a different fold serves as the assessment set while the other 9 train the model. The result: 10 independent performance estimates, averaged into one stable number.

The averaged ROC-AUC of ~0.84 is your working estimate of how well the attrition model generalizes — far more trustworthy than the ~0.98 you'd see if you naively evaluated on training data alone.

The Code

Three functions — vfold_cv(), fit_resamples(), collect_metrics() — replace the single fit() call when you want resampled performance:

# 1. Create 10 stratified folds from the training data
set.seed(502)
attr_folds <- vfold_cv(attr_train, v = 10, strata = Attrition)

attr_folds
#> #  10-fold cross-validation using stratification
#>  splits              id
#>  <list>              <chr>
#>  Fold01 ... Fold10

# 2. Fit the workflow across all 10 folds
attr_rs <- attr_wf %>%
  fit_resamples(resamples = attr_folds)

# 3. Collect averaged performance metrics
collect_metrics(attr_rs)
#> # A tibble: 2 × 6
#>   .metric  .estimator  mean     n std_err .config
#>   accuracy binary      0.874   10   0.011 Preprocessor1_Model1
#>   roc_auc  binary      0.837   10   0.015 Preprocessor1_Model1

Notice strata = Attrition in vfold_cv() — the same reason you used it in initial_split(). Each fold needs the same ~16% attrition rate as the full dataset, or your performance estimate will vary fold-to-fold for the wrong reasons.

Why ROC-AUC, Not Accuracy

The output gives you two metrics. For attrition data, ROC-AUC is the one that matters. Here's why:

Metric
What it measures
Attrition verdict
Accuracy
% of predictions that are correct
Misleading — a model that predicts "No" for everyone gets 84% accuracy on attrition data
ROC-AUC
Model's ability to rank leavers above stayers across all thresholds
Honest — 0.5 = random guessing, 1.0 = perfect, unaffected by class imbalance

A ROC-AUC of 0.837 means: pick any employee who left and any who stayed — the model ranks the leaver as higher risk 83.7% of the time. That's the number to bring to HR leadership.

The Pit of Success in Action fit_resamples() automatically applies the full workflow — recipe preprocessing and model fitting — inside every fold. You can't accidentally leak the assessment fold's statistics into the preprocessing. The correct practice is, again, the easiest path.
