Splitting Your HR Data

Protecting the test set — and why attrition data requires special care

The Data Budget

Before you write a single model, you have to make a decision about your data. Tidy Modeling with R frames this as a budget: you have a finite number of rows, and how you spend them determines whether your performance estimates are trustworthy.

The most important rule: set aside a test set before you do anything else, and don't touch it again until you're done building models. Every time you look at your test set — even to make a "small" decision — you spend some of that budget and inflate your eventual performance estimate.

Training set (80%)
Test (20%)
Sandbox for all modeling work — feature engineering, tuning, comparing models

The Attrition Imbalance Problem

IBM HR Attrition — the dataset you'll use throughout this course — has a real-world quirk: only about 16% of employees left. The other 84% stayed. This is typical for attrition data, and it creates a subtle trap.

If you split randomly, you might get a training set with 13% attrition and a test set with 22% — purely by chance. Your model trains on one distribution and gets evaluated on a different one. The performance number you report to HR leadership is measuring two different things.

What random splitting can produce
Full dataset
84% / 16%
Train (random)
87% / 13%
Test (random)
75% / 25%
Stayed Left (attrition)

The fix is one argument: strata = Attrition. This tells initial_split() to perform the 80/20 split within each class, then pool the results. Both sets end up with ~16% attrition, guaranteed.

The Code

Here is the complete splitting block for IBM HR Attrition. This is code you will run at the top of every People Analytics modeling script.

library(tidymodels)

# The attrition dataset lives in the modeldata package
# install.packages("modeldata") if needed
library(modeldata)
data(attrition)

# Set a seed so the split is reproducible
set.seed(502)

# 80% train / 20% test, stratified on the outcome
attr_split <- initial_split(attrition, prop = 0.80, strata = Attrition)
attr_train <- training(attr_split)
attr_test  <- testing(attr_split)

# Verify class balance was preserved in both sets
prop.table(table(attr_train$Attrition))
prop.table(table(attr_test$Attrition))

Run prop.table(table(...)) after splitting. You should see approximately No: 0.839, Yes: 0.161 in both sets — matching the original data's balance.

What You Now Have attr_train is your sandbox — all feature engineering, model building, and tuning happens here. attr_test sits untouched until you're ready to report a final honest performance number to leadership.

A Note on Time-Based HR Data

If your data has a time dimension — quarterly engagement surveys, monthly headcount snapshots — random splitting is wrong even with stratification. Randomly pulling a Q3 row into training while Q4 sits in test means your model "saw the future" during training.

For time-ordered data, use initial_time_split() instead: it puts earlier observations in training and recent ones in test. This matters most for workforce planning and headcount forecasting.

Gotcha initial_time_split() requires your data to be sorted by date first. Run arrange(date_column) before splitting, or you'll get a quietly incorrect result.
